# 요구사항 (간략)

이 문서는 FitDash mini의 화면/동작 요구사항 기준 문서입니다.
다른 문서에서는 이 문서의 상세 규칙을 반복하지 말고 참조만 유지합니다.

## 요구사항

- 플랫폼: iOS (Swift + SwiftUI)
- 초기 화면: 히트맵(Heatmap) 메인 화면이 앱 실행 시 기본으로 표시
- HealthKit
  - 읽기 권한: `HKObjectType.workoutType()` (읽기 전용)
  - 조회 대상: 러닝 워크아웃만 (`HKWorkoutActivityType.running`으로 필터)

### 뷰

- 히트맵 뷰(단일 메인): 연간 러닝을 한눈에 탐색하는 기본 화면
- 일자 상세 오버레이(히트맵 보조 인터랙션): 히트맵 셀 탭 시 해당 날짜의 러닝 세션 상세 확인
- 상단 타이틀은 필수 내비게이션 요소로 가정하지 않으며, 콘텐츠 중심 레이아웃을 유지합니다.

#### 히트맵 뷰 (GitHub 스타일 — 세로 배치)

- 연간(기본: 현재 연도) 그리드 히트맵을 세로로 배치한 뷰
- 연도 선택 탭을 제공하며, 사용자는 탭을 눌러 탐색할 연도를 전환할 수 있습니다.
- 셀은 연단위(한 덩어리)로 배치됩니다 — 월별로 셀이 분리되어 있지 않습니다.
- 화면에는 주(7일)가 세로로 배치된 그리드가 연속적으로 표시되며 세로 스크롤로 탐색합니다.
- 가로 레이아웃은 좌측 월 영역 / 중앙 7일 셀 영역 / 우측 주 합계 영역의 3열 구조를 유지합니다.
- 주 시작 요일: 월요일 고정
- 요일 헤더 표기는 `Mo Tu We Th Fr Sa Su`를 사용합니다.
- 각 날짜는 `RunDay`와 매핑된 정사각형 셀로 표시되고, 색상은 해당 날짜의 총 러닝 거리(km)에 따라 결정됩니다.
  - 셀 배경 정책: 배경색은 해당 날짜의 총 러닝 거리 강도만 표현합니다. 월 시작, 오늘, 선택 상태 같은 보조 의미를 배경색 변경으로 표현하지 않습니다.
  - 0 km: 빈 셀
  - 0 < km < 1: 매우 연한 색
  - 1 ≤ km < 5: 연한 색
  - 5 ≤ km < 10: 기본 녹색
  - 10 km 이상도 배경색은 `5 ≤ km < 10`과 같은 기본 녹색을 유지합니다.
- 셀 테두리 정책: 기본 테두리는 셀 경계를 구분하는 용도로만 사용합니다. 의미 상태는 서로 다른 레이어로 분리해 표현합니다.
  - 오늘: 전체 셀 외곽선 강조
  - 월 시작(매달 1일): 오늘과 다른 색의 별도 셀 테두리로 표시
  - 장거리 거리대는 외곽 테두리를 사용하지 않습니다.
  - 10 km 이상은 셀 내부의 작은 검은색 텍스트 배지로 표시합니다.
  - 거리대 표기 규칙: `숫자+` 형식으로 표시합니다. 예: `10+`, `20+`, `21+`, `30+`, `40+`, `42+`
  - 월 시작 표시는 텍스트나 코너 마커 대신, 오늘 테두리와 구분되는 색의 추가 외곽선으로 표시합니다.
  - 접근성 라벨에는 첫째 날 여부와 장거리 거리대 의미를 포함해 보조기기에서 구분되도록 합니다.
- 탭 가능한 셀은 실제 날짜가 있는 셀만 허용합니다(`nil` 패딩 셀 제외).
- 셀 탭 시 탭한 셀 근처에 경량 오버레이 카드(비모달)가 표시되고, 선택 날짜의 세부 정보를 표시합니다.
  - 카드 위치 정책: 가능한 경우 셀 위쪽에 노출하고, 위쪽 공간이 부족하면 셀 아래쪽에 노출합니다.
  - 카드 닫기 정책: 현재 선택된 날짜 셀을 다시 탭하면 카드가 닫힙니다.
  - 오버레이 헤더: 선택 날짜
  - 오버레이 본문: 해당 날짜 세션 목록(시작 시각, 거리 km, 페이스 /km)
  - 오버레이 요약: 총 거리, 세션 수
  - 해당 날짜 러닝이 없으면 빈 상태 문구를 표시합니다.
  - 오버레이가 열린 상태에서도 다른 날짜 셀을 연속 터치해 내용을 즉시 전환할 수 있습니다.
  - 스크롤이 시작되면(가로/세로) 오버레이 카드는 즉시 사라져야 합니다.
- 월 표시는 해당 월이 시작하는 날짜가 포함된 행의 왼쪽에 표시합니다.
- 월 라벨 계산은 범용 히트맵 표시 데이터 위에서 같은 주에 포함된 `1일`을 찾는 방식으로 결정합니다.
- 월 라벨은 `MMM + 월 합계`를 한 줄로 표시하며 줄바꿈 없이 표시합니다.
- 월 라벨이 없는 주(해당 주에 1일이 없음)도 월 영역의 최소 너비를 유지해야 합니다.
- 각 행(한 주 묶음)의 오른쪽에는 그 행에 속한 날짜들의 거리 합계를 표시합니다(행 합계).
- 셀 크기는 고정값을 유지합니다.
- 한 행의 높이는 셀 높이 기준으로 균일해야 하며, 보정용 `clear` 텍스트로 높이를 맞추지 않습니다.
- 월/주 영역 폭은 화면 가용 폭 비율 기반으로 배분합니다.
- 월/주 영역이 최소 폭까지 줄어든 뒤에도 전체 폭이 부족하면 가로 스크롤을 허용합니다.
- 연도 데이터 로딩/캐싱 정책: 앱 시작 시 HealthKit의 전체 러닝 워크아웃을 조회해 데이터가 있는 연도 탭을 구성합니다. 연도 전환은 캐시 데이터를 우선 사용하고, 선택 연도는 수동 새로고침으로 재조회할 수 있습니다.
- 뷰가 열리면(초기 진입 시) 화면은 자동으로 ‘오늘’ 위치로 스크롤됩니다. 데이터 로딩으로 내용이 갱신되더라도 중복 스크롤은 방지합니다.
- 히트맵 헤더 위에는 선택 연도의 누적 거리(연간 합계)를 표시합니다.
- 히트맵 헤더에는 새로고침 액션을 유지하며, `연 합계`, `월 합계`, `주 합계` 텍스트를 각각 기간 스냅숏 공유 진입점으로 사용합니다.
- 이미지 공유는 `연`, `월`, `주` 기간 스냅숏을 공통 UI로 처리하며, 날짜 상세 오버레이 상태와 무관하게 선택한 기간의 히트맵 카드 이미지 1장만 생성합니다.
- 연 스냅숏은 선택 연도 전체, 월 스냅숏은 선택한 월 전체, 주 스냅숏은 선택한 1주 7일만 표시합니다.
- 연 공유 진입은 선택 연도 데이터가 로드 완료 상태이고, 연간 총거리가 0km 초과이며, 연도 전환 pending 상태가 아닐 때만 활성화됩니다.
- 월/주 공유 진입은 해당 범위 합계가 0km 초과일 때만 동작합니다.
- 기간 스냅숏을 누르면 전용 미리보기 시트를 열고, 미리보기에서 시스템 공유 시트를 통해 이미지를 공유합니다.
- 공유용 이미지는 화면 스크린샷이 아니라 동일 기간 데이터로 다시 그린 정적 카드이며, 포함 정보는 `기간 제목`, `기간 합계`, `히트맵 그리드`, 최소 브랜딩(`FitDash`)으로 제한합니다.
- 공유용 이미지에는 월 라벨 텍스트와 주 합계 텍스트를 포함하지 않습니다.

### 모델 및 매니저

- 핵심 모델: `RunSession`, `RunDay` (일 단위 집계)
  - `RunSession`은 기본 표시 데이터(시작/종료 시각, 거리, 시간)와 함께 향후 상세 화면용 선택 데이터(이동 시간, 전체 시간, 고도 상승, 평균/최대 심박, 경로 좌표)를 담을 수 있습니다.
  - 현재 화면은 모든 상세 필드를 즉시 표시하지 않아도 되며, 기존 히트맵/목록 표시는 기본 거리/시간 데이터를 우선 사용합니다.
- 내부 보강 모델: `RunRecord`, `RunSource`
  - 화면 표시 전 여러 출처의 러닝 데이터를 `RunRecord`로 정규화하고, 기존 UI에는 `RunSession` 형태로 전달합니다.
  - Apple Health에 없는 외부/import 기록은 RunHeat 로컬 저장소에 보관할 수 있습니다.
  - HealthKit에는 계속 읽기 전용으로 접근하며, 외부/import 기록을 Apple Health에 쓰지 않습니다.
- Strava Bulk Export import
  - 개발 빌드에서는 Strava export archive를 풀어서 나온 `activities.csv`를 선택해 가져올 수 있습니다.
  - CSV의 `Activity ID`, `Activity Date`, `Activity Type`, `Distance`, `Moving Time`/`Elapsed Time`을 `RunRecord(source: .strava)`로 변환합니다.
  - `Activity Type`이 Run 계열인 행만 가져오고, 현재 단계에서는 `Filename`으로 연결되는 GPX 경로 파일은 파싱하지 않습니다.
- 공유 표시 모델: `HeatmapSnapshot` (`연/월/주` 기간의 공유용 합계/일별 거리 스냅숏)
- 매니저: `HealthKitManager`가 권한/조회와 HealthKit 결과 + 로컬 보강 기록 병합 책임

### Out of scope

- 별도 List 화면 제공 없음
- 별도 Calendar 화면 제공 없음
- 별도 상세 푸시 화면 제공 없음(셀 인접 오버레이 기반 상세만 사용)

## 필수 참고사항

- 시뮬레이터는 실제 HealthKit 데이터가 제한적이므로 개발 중에는 `MockHealthKitManager` 사용을 권장합니다.
  - 시뮬레이터 환경에서는 `Resources/` 폴더 내의 `running_workouts_*.json` 파일을 불러와 실제 데이터와 유사한 환경을 시뮬레이션합니다.
- HealthKit 권한은 필요한 항목만 요청하고, ContentView의 한 번만 인증 로직을 유지하세요.
- RunHeat 로컬 저장소는 Apple Health에 없는 외부/import 기록과 중복 제거용 메타데이터를 보관하는 보조 저장소입니다. HealthKit 전체 데이터를 복제하는 저장소로 사용하지 않습니다.
- 현재 HealthKit 읽기 권한은 `HKObjectType.workoutType()`만 요청하므로, 경로 좌표/심박/고도 등 상세 필드는 외부/import 데이터 또는 향후 별도 권한 확장 전까지 비어 있을 수 있습니다.
- **데이터 내보내기 및 테스트용 자료:** 
  - 개발 빌드(`#if DEBUG`)에서만 내보내기 버튼이 노출되며, 현재 연도의 데이터를 JSON으로 내보낼 수 있습니다.
  - 개발 빌드(`#if DEBUG`)에서만 Strava `activities.csv` import 버튼이 노출됩니다.
  - 내보낸 데이터에는 구간 페이스 계산을 위한 위치 데이터(`locations`)가 포함됩니다.
  - 내보낸 데이터는 개발 및 테스트를 위해 프로젝트 `Resources/`에 추가하여 Mock 데이터로 활용할 수 있습니다.
  - 출시용 빌드(Release)에는 내보내기 기능 및 관련 테스트용 리소스가 포함되지 않도록 관리합니다.
  - **한계점:** Garmin과 같은 타사 기록 앱 중 일부는 `HKWorkoutRoute`(경로 데이터)를 HealthKit에 직접 쓰지 않는 경우가 있습니다. Apple 피트니스 앱을 사용하지 않은 세션 중 경로 데이터가 없는 경우, 앱 내에서 1km 구간별 페이스(Splits) 계산이 불가능하며 이 경우 상세 뷰에서 스플릿 정보가 표시되지 않습니다.
