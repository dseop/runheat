# 유지보수 점검

이 문서는 개발 현황 파악과 문서 정리를 반복 가능한 작업으로 편입하기 위한 체크리스트입니다. 상세 기능 규칙은 `docs/requirements.md`, 코드 구조 설명은 `docs/development.md`, 출시 문구는 `docs/app_store.md`를 기준으로 합니다.

## 실행 시점

- 기능 단위 작업 완료 후
- 커밋 또는 PR 직전
- TestFlight 또는 App Store 제출 전
- 큰 기능 변경이 없어도 월 1회

## 점검 순서

1. `git status --short`로 작업 범위와 미추적 파일을 확인한다.
2. `docs/requirements.md`와 실제 화면/동작이 맞는지 확인한다.
3. 코드 구조나 책임 경계가 바뀌었으면 `docs/development.md`를 갱신한다.
4. 출시 문구, 심사 메모, 릴리즈 노트가 필요하면 `docs/app_store.md`를 갱신한다.
5. 문서 위치나 성격이 바뀌었으면 `docs/README.md`와 `AGENTS.md`의 인덱스를 갱신한다.
6. `.DS_Store`, 임시 파일, 로컬 산출물, 중복 문서가 프로젝트에 섞였는지 확인한다.

## 판단 기준

- 요구사항의 단일 기준은 `docs/requirements.md`다.
- 코드 구조와 유지보수 기준은 `docs/development.md`에 둔다.
- App Store Connect 문구와 릴리즈 노트는 `docs/app_store.md`에 둔다.
- 기능 아이디어나 보류 항목은 계획 문서로 유지하되, 구현 기준처럼 쓰지 않는다.
- 오래된 계획 문서는 삭제하기 전에 `참고용`, `보류`, `폐기 후보` 중 하나로 상태를 명확히 한다.

## 정리 대상 예시

- 실제 코드와 맞지 않는 구조 문서
- 다른 문서와 역할이 중복되는 문서
- 현재 기능 기준을 오해하게 만드는 오래된 초안
- `.DS_Store` 같은 OS 메타데이터
- 추적할 필요가 없는 일회성 로컬 산출물
